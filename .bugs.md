# Bug worklist (from rules-vs-code review, 2026-07-12)

Compared against Mark III v2.05 rulebook + card text in material.inc.php.
Reported-bugs list from BGA studio not yet cross-referenced (needs login).

## Rule bugs — high impact

- [ ] **Event-draw check uses actions remaining, not actions taken** — `pass()` burglebros.game.php:4326-4327. `actions_remaining >= 2` only equals "took ≤2 actions" when turn had exactly 4. Wrong with Rollerskates (6), Mirror (3), Heads Up (5), Invisible Suit (5), Stamp variant same flaw. Fix: track actions granted at turn start, compare actions used.
- [ ] **Gymnastics floor connection — rules-edition conflict, decide** — `walkwayIsAdjacent()` burglebros.game.php:1089-1105. Code: walkway ↔ tile *above*, both directions — matches 2016 FAQ ruling ("treated as the Stairs tile for that floor", zenxacred). Mark III v2.05 rulebook p16 says instead: move into Walkway from tile *below* it. Code follows old ruling; rulebook is newer. Probably leave as-is or match rulebook deliberately.
- [ ] **Walkway-down doesn't ignore entry effects** — rulebook p22: moving downstairs from revealed Walkway ignores entering effects (Deadbolt cost, Fingerprint alarm...). Currently goes through full `handleTileMovement()` (:2047).
- [ ] **Walkway fall costs Stealth if guard in landing tile** — burglebros.game.php:2148 calls `handlePlayerEnteredGuardSight` after fall. Rulebook p21: fall doesn't count as entering, no stealth loss.
- [ ] **Crash event missing "if no Alarm is currently active" gate** — burglebros.game.php:2408-2412 (Change of Plans has the check; Crash doesn't).
- [ ] **Throw Voice missing "if no Alarm is currently active" gate** — burglebros.game.php:2934-2949.
- [ ] **Switch Signs missing alarm clause + arrival effects** — burglebros.game.php:2512-2531. Rulebook p16: if alarms active → guard to closest alarm, deactivate, redraw destination. Also no stealth loss / camera check / alarm deactivation on guard's new tile (TODO comment admits it).
- [ ] **Freight Elevator top-floor escape not implemented** — burglebros.game.php:2420-2433. Rulebook p15: on top floor, immediately escape if team holds all loot.
- [ ] **Operator precedence in `triggerAlarm` suppression** — burglebros.game.php:2232. `a || b || emp && (!skip && hacker)` — EMP alone never suppresses; hacker only applies when EMP active. Reachable: Dynamite (:2785), Juicer alarms during active E.M.P.
- [ ] **Motion exit check wrong** — burglebros.game.php:2169: `$motion_entered && $motion_bit` should be bitwise `&`. Also `motionTileEntered` is global per turn, not per player (Rook-moved player pollutes it, :2122).

## Rule bugs — edge cases

- [ ] **Peterman Drill + Stethoscope** — `applyDieRoll()` (:1708-1711) recomputes safe tile as *player's* tile → wrong floor for cracked/open tokens. Also `performSafeDiceRoll` `nextState('cardChoice')` (:1689) from cardChoice state has no such transition (states.inc.php:132) → server exception.
- [ ] **`nextPatrol` force + 2+ equally-close alarms** — burglebros.game.php:882-903: die lands on freshly drawn patrol card instead of an alarm. Reachable when guard first revealed on a floor that already has multiple alarms.
- [ ] **Multi-alarm choice lost in loops** — Shift Change overwrites `$choose_alarm` per floor (:4710); Shoplifting overwrites `$special_choice` per lab (:2509). Only last survives; stale `specialChoiceArg` bleeds into next guard move (:1176).
- [ ] **Heads Up escaped-player skip loop doesn't advance** — burglebros.game.php:2447-2453: `getPlayerAfterCustom()` returns same player each iteration.
- [ ] **Dead Drop can force both Gold Bars onto one player** — burglebros.game.php:2413-2419, no gold-bar check (trades and takeCards have one).
- [ ] **Hacker moved by Rook still triggers Fingerprint** — `hackerDoesNotTrigger()` (:1965-1983) keys off current player, checks hacker position before move. Jammer is non-optional.
- [ ] **Keycard + Drill interpretation** — :1660-1668 lets advanced Peterman crack remotely if he holds Keycard; RAW says holder must be in Safe tile. Decide/document house ruling.

## Server-side validation / robustness

- [ ] **Trade validation is dead code** — `proposeTrade()` burglebros.game.php:4054-4055: `array_merge` result discarded → type/ownership loop iterates nothing; ownership check compares `$card['location']` instead of `location_arg` (:4059).
- [ ] **`escape()` never checks position** — :4379-4392, only `checkWin()`; on-stairs/top-floor requirement only in UI (`canEscape`).
- [ ] **`move()` trusts client-supplied `context`** — burglebros.action.php:73; `context=rook1` skips Deadbolt/Laser costs and alarms (:2074, :2107).
- [ ] **`confirmTakeCards` crashes on zero picked cards** — `$new_card` used outside loop (:4238); donuts reset only inspects last card taken.
- [ ] **`confirmRookMove` missing `checkAction`** — burglebros.game.php:4257.
- [ ] **`zombieTurn` uses nonexistent `zombiePass` transition** — :4936; player quitting mid-game throws and stalls table.
- [ ] **Hidden info leak: deck contents in `getAllDatas`** — `gatherCardData()` (:614) ships full event/tool/loot/patrol deck cards incl. `type_arg` (draw order) to client.
- [ ] Trivial: `getFloorCount() == 4` should be `getSquareSize() == 4` (:814, :2559) — harmless today (25 > any 4x4 path).

## Client (JS) trivia

- [ ] **Patrol tooltip "floor" is garbage** — `patrolCardHtml()` burglebros.js:1618: `card['location'][5]` — patrol cards live in `patrolN_deck`/`patrolN_discard`, so char 5 is always the letter 'l'. The tooltip div id renders as `patrol_tooltip_l` on every floor (duplicate ids). Harmless today (id never referenced); fix by passing the floor in — both callers know it.
- [ ] **`canCancelCardChoice` matches type_arg across card types** — burglebros.js:1063: crystal-ball (tool 3), spotter1 (char 17), spotter2 (char 18) are matched on type_arg alone, so any card sharing those type_args (e.g. hacker1 = character 3) is also uncancellable if it ever reaches cardChoice.

## Needs investigation

- [ ] **Guard clockwise tie-break** — `breakTie()` (:1520-1552) only compares first two paths after `sort()`; 3+ equally short paths may pick wrong. `clockwise_mappings` table needs simulation to validate. Most likely source of guard-path bug reports.
- [ ] **Pathfinding perf** — `findShortestPathClockwise()` (:1554-1592) enumerates ALL simple paths via DFS; on lightly-walled 5x5 Fort Knox floor could blow up / timeout. Profile.
